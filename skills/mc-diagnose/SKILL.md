---
name: mc-diagnose
description: 营销全链路诊断引擎。用户说"效果不好"、"最近不行了"、"哪里出了问题"时触发——从模糊症状出发，通过八维框架定位根因，输出 ICE 优先级排序的修复建议和对应技能路由。不需要用户先提供数据，诊断过程自己找。
---

## 触发条件

当用户提到以下任意情形时使用本技能：

- "效果不好" / "最近不行了" / "出了问题" / "怎么回事"
- "广告不行了" / "投放效果变差" / "花钱没效果"
- "不知道哪里出了问题" / "感觉哪里不对"
- "帮我诊断一下" / "全面检查一下" / "做个审计"
- "销量下降了但不知道为什么" / "数据变差了"
- "紧急！XX 出问题了" / "救命" / "急救"
- "上次修了但又不行了" / "老问题又出现了"

---

## 与其他技能的关系

| | mc-diagnose | mc-analytics |
|---|---|---|
| 输入 | 模糊症状（"效果不好"） | 具体数据（"帮我看这些数字"） |
| 起点 | 不知道问题在哪 | 知道要分析什么 |
| 过程 | 从症状反向追踪到环节 | 从数据正向解读到结论 |
| 产出 | 根因定位 + 修复优先级 + 技能路由 | 数据解读 + 决策建议 |
| 协作 | 诊断完成后推荐执行 mc-analytics 深挖或其他技能修复 | 分析时发现系统性问题可升级为 mc-diagnose |

mc-diagnose 回答"到底哪里出了问题"；mc-analytics 回答"这个数字什么意思"。

---

## 核心原则

**症状不是病因。** 用户说的"广告效果差"可能是素材问题、可能是定向问题、可能是落地页问题、甚至可能是产品问题。诊断的价值是穿透表象，找到真正该修的环节。

- 不接受用户的自我诊断 → 用户说"我觉得是素材问题"，仍然跑完整排查
- 不在单一环节停留 → 即使某环节明显异常，也快速扫完其他维度
- 所有结论附置信度 → "80% 确信是落地页问题" 比 "是落地页问题" 更诚实
- 修复建议必须路由到具体技能 → 不是"优化内容"，是"执行 mc-content 重做小红书内容矩阵"

---

## 工作方式

1. 收集症状描述（用户的原话 + 追问 1-2 个关键问题）
2. 读取 campaign 上下文（`brand.md`、`strategy.md`、`analytics.md`）
3. 读取 `memory/brand-memory.md` 中的历史诊断记录（如有）
4. 按八维框架执行诊断
5. 产出写入 `campaigns/{project-slug}/diagnose.md`

## 五种诊断模式

收到请求后，先判断用什么模式：

| 模式 | 触发条件 | 执行范围 | 时间预期 |
|------|---------|---------|---------|
| 🔍 全面审计 | "帮我全面检查一下" / 首次合作 / 无明确症状 | 八维全扫 | 完整 |
| 🎯 专项深挖 | 用户指定了某个环节（"广告效果差"） | 聚焦 1-2 维 + 关联维度快扫 | 中等 |
| 🚑 急救 | "紧急" / "救命" / 数据断崖下跌 | 快速定位 → 止血方案 → 跳过非紧急维度 | 快速 |
| 🔄 跟进 | "上次修了但还是不行" / 有历史诊断记录 | 从上次结论出发，验证修复效果 + 扩展排查 | 中等 |
| 🚨 危机 | 现金流告急 / PR 事件 / 平台封号 | 按危机类型走专项协议（见下方） | 快速 |

## 八维诊断框架

### 维度概览

```
┌─────────────────────────────────────────────┐
│              营销全链路健康度                   │
├──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┤
│ 流量  │ 转化  │ 客单  │ 复购  │ 品牌  │ 渠道  │ 内容  │ 运营  │
│      │      │      │      │      │      │      │      │
│ 来的  │ 买不  │ 买多  │ 回不  │ 记不  │ 钱花  │ 说的  │ 做的  │
│ 人够  │ 买    │ 少    │ 回来  │ 记得  │ 对不  │ 对不  │ 稳不  │
│ 不够  │      │      │      │      │ 对    │ 对    │ 稳    │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

### 维度 1：流量（来的人够不够）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 总流量趋势 | 周环比波动 <15% | 持续 3 周下降 |
| 流量来源结构 | 付费/自然/直接各有占比 | 单一来源 >80% |
| 流量质量 | 跳出率 <60%，停留 >60s | 跳出率 >80% 或停留 <20s |
| 新老访客比 | 新客 40-70% | 新客 >90%（无复访）或 <20%（无新增）|

**关联维度**：流量异常 → 先查渠道（是不是某个渠道出了问题）→ 再查内容（是不是素材/内容吸引力下降）

### 维度 2：转化（买不买）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 整体 CVR | 达到品类基准 | 低于基准 50% |
| 漏斗断裂点 | 各环节衰减均匀 | 某环节流失 >70% |
| 加购→支付 | 弃购率 <70% | 弃购率 >85% |
| 移动端 vs 桌面端 | 差异 <30% | 移动端 CVR 不到桌面一半 |

**关联维度**：转化异常 → 查流量质量（是不是来的人不对）→ 查客单（是不是价格吓跑人）

### 维度 3：客单价（买多少）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| AOV 趋势 | 稳定或缓升 | 持续下降 |
| 件单价 vs 客单价 | 客单 > 件单（有搭配购买）| 客单 ≈ 件单（无连带）|
| 促销依赖度 | 促销订单 <40% | 促销订单 >70%（不打折不买）|
| 价格带分布 | 集中在利润款 | 集中在引流款 |

**关联维度**：客单异常 → 查产品组合（是不是缺高价值 SKU）→ 查品牌（品牌溢价力不足）

### 维度 4：复购（回不回来）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 30/60/90 天复购率 | 达到品类基准 | 低于基准 50% |
| 首购→二购 间隔 | 品类合理周期内 | 远超品类周期 |
| 客户流失曲线 | 平缓衰减 | 首购后断崖流失 |
| 老客贡献占比 | 30-60% | <15%（全靠拉新）|

**关联维度**：复购异常 → 查运营（购后触达是否到位）→ 查产品（是否有复购属性）

### 维度 5：品牌（记不记得）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 品牌搜索量 | 月环比增长 | 持续下降或停滞 |
| 品牌词占自然流量比 | >20% | <5% |
| 品牌提及情感 | 正面 >70% | 负面 >30% |
| 品类关联度 | 提到品类会想到品牌 | 品牌认知为零 |

**关联维度**：品牌异常 → 查内容（品牌故事是否在传播）→ 查渠道（是不是只做效果广告没做品牌）

### 维度 6：渠道（钱花对不对）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 渠道 ROAS 分布 | 多个渠道 ROAS >1 | 只有 1 个渠道盈利 |
| 渠道集中度 | 最大渠道 <50% | 单渠道 >80%（高风险）|
| 新渠道测试 | 持续有测试 | 6 个月无新渠道尝试 |
| 渠道间协同 | 有交叉引流 | 各渠道孤岛运营 |

**关联维度**：渠道异常 → 查内容（各渠道内容是否适配）→ 查流量（是不是某渠道流量质量差）

### 维度 7：内容（说的对不对）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 内容互动率 | 高于行业均值 | 低于行业均值 50% |
| 内容生命周期 | 持续有自然流量 | 发布后 3 天即无流量 |
| 内容与品牌一致性 | 调性统一 | 各平台人格分裂 |
| UGC 比例 | 有用户自发创作 | 全靠品牌自产 |

**关联维度**：内容异常 → 查品牌（品牌故事是否清晰）→ 查流量（内容分发是否到位）

### 维度 8：运营（做的稳不稳）

| 检查项 | 健康信号 | 异常信号 |
|--------|---------|---------|
| 客服响应 | <4 小时 | >24 小时 |
| 退货率 | 品类合理范围 | 远高于行业 |
| 库存周转 | 健康周转 | 积压或断货 |
| 自动化覆盖 | 关键流程已自动化 | 全靠人工 |

**关联维度**：运营异常 → 查复购（是不是售后体验差导致不回来）→ 查转化（是不是发货慢导致取消）

## 三阶段诊断流程

### 阶段 1：症状拆解

把用户的模糊描述转化为可检测的信号：

```markdown
## 症状拆解

**用户原话：** "{用户说的话}"

**拆解为可检测信号：**

| 可能含义 | 对应维度 | 需要检查的指标 |
|---------|---------|--------------|
| {解读 1} | {维度} | {具体指标} |
| {解读 2} | {维度} | {具体指标} |
| {解读 3} | {维度} | {具体指标} |

**优先排查顺序：** {维度 A} → {维度 B} → {维度 C}
（理由：{为什么先查这个}）
```

### 阶段 2：数据检索

根据排查顺序，向用户收集或从已有文件中提取关键数据：

**信息收集规则：**
- 已有 `analytics.md` / `report.md` → 直接读取，不重复追问
- 用户能提供数据 → 逐维度收集关键指标
- 用户无法提供 → **用行业基线做假设性诊断**，标注 `⚠️ 基于行业基线，非实际数据`
- 最多追问 2 轮，之后用已有信息给出最佳判断

### 阶段 3：根因判定

```markdown
## 诊断结论

### 根因定位

| 排序 | 维度 | 问题 | 置信度 | 证据 |
|------|------|------|--------|------|
| 1 | {维度} | {具体问题描述} | {高/中/低} | {支撑判断的数据或信号} |
| 2 | {维度} | {具体问题描述} | {高/中/低} | {支撑判断的数据或信号} |
| 3 | {维度} | {具体问题描述} | {高/中/低} | {支撑判断的数据或信号} |

### 因果链

{用 → 箭头描述问题之间的因果关系}

示例：品牌认知下降 → 自然流量减少 → 付费获客成本上升 → ROAS 下降 → 用户感知"效果不好"

### 修复建议（ICE 排序）

| 优先级 | 修复动作 | Impact | Confidence | Ease | ICE | 执行技能 |
|--------|---------|--------|------------|------|-----|---------|
| 1 | {具体动作} | {1-10} | {1-10} | {1-10} | {总分} | mc-{skill} |
| 2 | {具体动作} | {1-10} | {1-10} | {1-10} | {总分} | mc-{skill} |
| 3 | {具体动作} | {1-10} | {1-10} | {1-10} | {总分} | mc-{skill} |

### 下一步

1. **立即执行：** {ICE 最高的动作}，调用 mc-{skill}
2. **本周完成：** {第 2-3 优先级的动作}
3. **持续监控：** {需要观察的指标和复查时间}
```

## 危机模式协议

### 🚨 现金危机

触发信号：用户提到"现金流紧张" / "快没钱了" / "亏损严重" / "要倒闭了"

**优先级重排：**
1. 止血 → mc-analytics（快速定位最大亏损源）
2. 清库存 → mc-automation（设计清仓序列）+ mc-copy（促销文案）
3. 砍低效 → mc-analytics（识别 ROAS <1 的渠道，立即停投）
4. CRO 快修 → mc-dtc（结账流程优化，提升现有流量转化）

**暂缓项目：** 品牌建设、内容矩阵、新渠道测试

### 🚨 PR 危机

触发信号：用户提到"被骂了" / "上热搜了" / "品牌出事了" / "负面新闻"

**优先级重排：**
1. 评估 → mc-review（合规检查 + 风险定级）
2. 社群安抚 → mc-community（私域用户沟通 SOP）
3. 舆情追踪 → mc-monitor（竞品和行业动态监控）
4. 内容修正 → mc-content（调整内容策略，暂停争议性内容）

**暂缓项目：** 达人投放、付费广告、新品发布

### 🚨 平台封号 / 限流

触发信号：用户提到"被封了" / "限流了" / "审核不过" / "违规"

**优先级重排：**
1. 合规修复 → mc-review（定位违规点 + 申诉方案）
2. 渠道分散 → mc-diagnose（评估渠道集中度风险）
3. 流量替代 → mc-seo / mc-geo（启动有机流量建设）
4. 私域导流 → mc-community（将公域粉丝导入私域降低平台依赖）

## 诊断记忆

每次诊断完成后，将关键结论写入 `memory/brand-memory.md` 的诊断章节：

```markdown
## 诊断记录

### {日期} — {诊断模式}

**症状：** {一句话}
**根因：** {一句话}
**修复状态：** 待执行 / 执行中 / 已修复 / 复发
**关联技能：** mc-{skill}
```

下次诊断时读取历史记录，避免重复排查已知问题，重点关注"复发"和"执行中"的项目。
